# AI卡牌桌面项目 - 技术交接文档 v8.2.0

**文档版本**: 1.1
**更新日期**: 2024-05-23
**目标**: 本文档旨在清晰地阐述“AI卡牌桌面”插件在完成V8架构重构后的当前状态、面临的主要技术挑战以及下一阶段的开发路线图。

---

## 1. 项目现状 (Project Status)

经过近期的密集迭代，项目已成功迁移至一个现代化、模块化的前端架构，其设计灵感源自“手机模拟器”插件的成功实践。目前的核心成果包括：

-   **稳健的架构**: 采用了基于 ES6 模块的清晰文件结构 (`dataHandler`, `commandParser`, `ui`, `events` 等)，并通过 `fetch` API 实现了可靠的异步UI加载，从根本上解决了先前版本中存在的“静默启动失败”问题。
-   **完整的游戏循环**: 现已支持完整的Roguelike游戏循环。玩家在开始时会进入**难度选择界面**，选择后将初始化玩家数据并生成新地图。游戏中可以通过设置菜单**重置挑战**，或**尝试逃跑**。
-   **原子化的世界书操作**: `dataHandler` 模块现在使用 `TavernHelper_API.updateWorldbookWith` 来执行所有世界书的更新。这种原子化的批量读写模式，彻底解决了因异步竞争条件导致的数据不一致或丢失问题（例如之前反复出现的发牌失败）。
-   **强大的指令解析器**: 重写了 `commandParser.js`，使其能够健壮地解析包含复杂内嵌JSON对象的AI指令，这是实现 `[Game:Start]` 和 `[Game:Function]` 指令正确执行的关键。
-   **核心游戏逻辑已初步打通 (AI -> 前端)**:
    1.  AI能够成功发出 `[Game:Start]` 指令来初始化一场牌局。
    2.  前端能够正确解析该指令，更新敌人数据 (`sillypoker_enemy_data.json`) 和公共游戏状态 (`sillypoker_current_game_state.json`)。
    3.  AI能够成功发出 `[Game:Function, type:发牌]` 指令。
    4.  前端的 `dataHandler` 能够正确响应，从牌堆中为玩家和敌人发牌，并原子化地更新 `sillypoker_player_cards.json`、`sillypoker_enemy_data.json` 和 `sillypoker_current_game_state.json`。
-   **基础UI渲染**: UI可以根据世界书中的数据，正确渲染出牌桌、玩家手牌、敌人牌背以及地图等基本界面。

**简而言之，项目的底层数据流和AI指令的接收/执行链路已经基本稳固，为接下来的功能开发奠定了坚实的基础。**

## 2. 当前面临的主要挑战与技术债 (Current Challenges & Technical Debt)

尽管底层架构已经稳定，但目前项目在功能完整性上仍存在巨大差距。核心挑战集中在**“玩家无法与游戏世界交互”**上。

### 2.1 **核心挑战：玩家操作循环中断 (Player Action Loop is Broken)**

这是当前 **最高优先级** 的问题。玩家虽然能看到UI，但无法执行任何游戏内操作。

-   **问题描述**: 游戏界面上的所有玩家操作按钮（如下注、过牌、弃牌）以及地图上的节点，其事件监听器虽然存在，但并未实现将玩家操作反馈给AI的完整逻辑。
-   **技术缺失**:
    1.  **操作暂存机制**: 需要一个类似于“手机模拟器”的暂存系统 (`stagedPlayerActions`) 来收集玩家在一个回合内执行的所有操作。
    2.  **提交与格式化**: 缺少一个核心函数（例如 `commitStagedActions`），用于将暂存的操作转换为AI可以理解的、格式化的系统提示，例如 `(系统提示：玩家选择了“跟注”，下注100筹码。)`。
    3.  **触发AI**: 缺少在提交操作后，自动调用 `SillyTavern_API.generate()` 来触发AI响应的机制。

### 2.2 AI决策逻辑尚未实现 (AI Decision Logic is Rudimentary)

目前，AI仅扮演了“发牌员”的角色，并未真正成为“对手”。

-   **问题描述**: AI在发牌后便会停止行动，因为它缺少被告知需要进行决策的明确提示，也没有现成的逻辑来读取和理解当前的游戏状态。
-   **技术缺失**:
    1.  **AI提示工程**: 需要设计和优化一系列系统提示，引导AI在轮到它行动时，去主动“思考”并“查阅” `sillypoker_current_game_state.json` 的内容。
    2.  **行动指令实现**: AI需要根据其 `play_style`（如“激进型”、“谨慎型”）和当前牌局状态，生成如 `[Action:Bet]`、`[Action:Check]` 等具体行动指令。前端对这些 `Action` 指令的解析和执行逻辑尚未编写。

### 2.3 UI/UX 处于原型阶段 (UI/UX is Prototypical)

-   **问题描述**: 当前UI仅为功能性占位符，缺乏动态反馈和沉浸感。
-   **技术缺失**:
    1.  **状态反馈**: 缺少基于游戏状态的UI变化，例如在非玩家回合禁用操作按钮、高亮当前行动的玩家等。
    2.  **动画与音效**: 缺少发牌、下注、胜负结算等关键操作的视觉动画和音效，极大地影响了游戏体验。
    3.  **信息展示**: 缺少对道具、状态效果、特殊规则的清晰可视化展示。

## 3. 下一步开发路线图 (Next Steps & Development Roadmap)

以下是建议的、按优先级排序的开发计划：

### **Phase 1: 闭合核心玩法循环 (Priority: CRITICAL)**

**目标**: 让一场基础的牌局能够完整地进行下去。

1.  **实现玩家操作暂存**:
    -   在 `AIGame_State` 中添加 `stagedPlayerActions` 数组。
    -   为游戏桌上的“下注”、“过牌”、“弃牌”按钮和地图节点添加事件监听器 (`events.js`)。
    -   当玩家点击时，将对应的操作（如 `{type: 'call', amount: 100}`）`push` 到 `stagedPlayerActions` 数组中，并**立即更新UI**以提供即时反馈（例如显示玩家已下注的筹码）。

2.  **实现操作提交机制**:
    -   创建一个 `commitStagedActions` 函数 (`dataHandler.js`)。
    -   该函数会遍历 `stagedPlayerActions`，生成一个单一的、对AI友好的系统提示字符串。
    -   调用 `/setinput` 和 `generate()` 将提示发送给AI。
    -   清空 `stagedPlayerActions` 数组。
    -   在UI上添一个“结束回合”或“提交操作”的按钮来触发此函数。

3.  **实现AI行动指令的解析**:
    -   在 `dataHandler.js` 中，为 `[Action:Bet]`, `[Action:Check]` 等指令编写处理逻辑。
    -   这些逻辑会修改 `sillypoker_current_game_state.json` 和 `sillypoker_enemy_data.json` 中的数据（如筹码、彩池）。

### **Phase 2: 提升AI智能与游戏深度 (Priority: HIGH)**

**目标**: 让AI成为一个真正的、有策略的对手。

1.  **完善AI提示**: 优化默认角色卡或系统提示，明确指示AI在收到 `(系统提示: ...)` 且轮到它行动时，必须分析 `current_game_state` 并输出一个 `[Action:...]` 指令。
2.  **扩展游戏逻辑**: 丰富 `[Game:UpdateState]` 指令的使用，让AI能够主动更新公共牌、彩池等信息。
3.  **实现道具与状态系统**: 在 `dataHandler` 中编写逻辑，使AI能够识别并响应玩家和敌人数据中的 `inventory` 和 `status_effects`。

### **Phase 3: 全面优化UI/UX体验 (Priority: MEDIUM)**

**目标**: 将粗糙的原型打磨成一个引人入胜的游戏界面。

1.  **添加动画效果**: 为发牌、收牌、筹码移动、胜负结算等添加流畅的CSS或JS动画。
2.  **集成音效**: 为关键操作（点击、下注、洗牌、警告等）添加音效。
3.  **完善状态显示**: 根据 `current_turn` 禁用/启用按钮，高亮当前玩家，清晰地展示游戏日志。

### **Phase 4: 扩展游戏多样性 (Priority: LONG-TERM)**

**目标**: 验证并发挥框架的可扩展性。

1.  **添加新游戏模块**: 创建一个新的游戏渲染文件（如 `blackjackRenderer.js`），并扩展 `stateGenerator.js` 以支持新的 `game_type`。
2.  **设计新的AI对手**: 引导AI利用其 `play_style` 和 `ability` 属性，创造出行为模式各异的精英和Boss级对手。

## 4. 总结 (Conclusion)

“AI卡牌桌面”项目已经成功地搭建了一个稳固且可扩展的底层架构。当前的瓶颈在于玩家与游戏世界的交互是单向的。一旦**Phase 1**完成，核心玩法循环得以闭合，整个框架的潜能将被释放，后续的AI智能化和内容扩展将变得水到渠成。
